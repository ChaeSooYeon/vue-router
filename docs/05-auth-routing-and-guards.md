# 네비게이션 가드와 라우팅 흐름 정리

## 1. 네비게이션 가드 개념과 사용

### 네비게이션 가드

- 네비게이션 가드는 라우트 이동이 일어나기 전에 실행되는 검사 로직이다.
- 이번 프로젝트에서는 `router.beforeEach()`를 사용해 페이지 이동 전 로그인 여부를 확인했다.
- 인증이 필요한 페이지에 비로그인 상태로 접근하면 원하는 페이지로 바로 가지 못하고 로그인 페이지로 우회시킬 수 있다.
- 즉, 사용자가 페이지에 "들어간 뒤" 막는 것이 아니라, 라우터가 "들어가기 전" 이동을 제어하는 방식이다.

#### 전역 Resolve 가드

- router.beforeEach와 유사하게 모든 내비게이션에서 트리거되지만, resolve 가드는 내비게이션이 확정되기 직전, 모든 컴포넌트 내 가드와 비동기 라우트 컴포넌트가 해결된 후에 호출된다.
- 예로 beforeEach로 페이지에 대한 접근 자체를 검사하고, 이동 하면, beforeResolve로 카메라 접근 권한을 확인하는 방식이다.

```js
router.beforeResolve(async (to) => {
  if (to.meta.requiresCamera) {
    try {
      await askForCameraPermission();
    } catch (error) {
      if (error instanceof NotAllowedError) {
        // ... 오류를 처리한 후 내비게이션 취소
        return false;
      } else {
        // 예기치 않은 오류, 내비게이션 취소 및 오류를 전역 핸들러로 전달
        throw error;
      }
    }
  }
});
```

- router.beforeResolve는 사용자가 페이지에 진입할 수 없는 경우 데이터를 가져오거나 기타 작업을 피하고 싶을 때 이상적인 위치이다.

### 라우트 메타 정보

- 라우트 객체의 `meta`는 화면 이동 정책을 설명하는 보조 정보다.
- 이번 작업에서는 `meta.requiresAuth`를 사용해 "이 페이지는 로그인 후에만 접근 가능"하다는 규칙을 라우트에 직접 적었다.
- 이렇게 하면 가드에서 특정 경로 문자열을 하드코딩하지 않고도 보호 페이지 여부를 판단할 수 있다.

### 리다이렉트

- 리다이렉트는 어떤 페이지에 바로 들어가지 못할 때 다른 페이지로 먼저 보내는 처리다.
- 이번 작업에서는 비로그인 사용자가 `/my-page`에 접근하면 `/login`으로 보낸 뒤, 로그인 성공 시 다시 원래 목적지로 돌려보내는 흐름을 만들었다.
- 이 정보는 `query.redirect`에 저장해서 로그인 화면과 회원가입 화면에서도 이어서 사용할 수 있게 했다.

### 접근 사유 전달

- 단순히 로그인 페이지로 보내는 것만으로는 사용자가 왜 이동됐는지 알기 어렵다.
- 그래서 `query.reason` 값을 함께 전달해 "인증이 필요해서 이동된 것인지", "로그아웃 후 이동된 것인지"를 화면에서 설명할 수 있게 했다.

## 2. 인증과 네비게이션 가드

- 네비게이션 가드와 라우팅 흐름을 확인하기 위해 실제 백엔드 구현 대신 간단한 목업 API를 사용했다.

### 보호 페이지 접근 제어

- `meta.requiresAuth`가 있는 라우트는 전역 네비게이션 가드에서 로그인 여부를 검사하도록 정리했다.
- 이유는 인증이 필요한 페이지를 각 컴포넌트에서 따로 막는 것보다, 라우터 진입 지점에서 공통 규칙으로 처리하는 편이 더 명확하기 때문이다.

### 리다이렉트 정보 전달 방식 통일

- 보호 페이지로 접근하다가 로그인 페이지로 이동할 때 목적 경로를 `query.redirect`로 넘기도록 통일했다.
- 이유는 이동 흐름을 URL에 드러나게 만들면 디버깅과 확인이 쉽고, 로그인과 회원가입 화면에서도 같은 방식으로 이어받을 수 있기 때문이다.

### `history.state` 대신 `query`를 사용한 이유

- 처음에는 `history.state?.redirect` 방식도 사용할 수 있지만, 이 값은 브라우저 히스토리 엔트리에만 들어가서 현재 어떤 리다이렉트 정보가 실려 있는지 바로 보기가 어렵다.
- 반면 `route.query.redirect`는 URL에 직접 드러나기 때문에 현재 목적 경로를 확인하기 쉽고, 로그인 화면에서 회원가입 화면으로 이동할 때도 같은 방식으로 전달하기 편하다.
- 이번처럼 네비게이션 가드와 인증 라우팅 흐름을 확인하는 구조에서는 `query` 방식이 더 직관적이고 디버깅에도 유리하다.
- 다만 URL을 최대한 깔끔하게 유지하거나 목적 경로를 주소창에 노출하고 싶지 않은 경우에는 `history.state` 방식도 선택할 수 있다.

### 로그인 후 원래 페이지로 복귀

- 로그인 성공 시 기본적으로 `myPage`로 가되, `query.redirect` 값이 있으면 원래 요청한 경로로 우선 복귀하도록 만들었다.
- 이유는 사용자의 흐름이 끊기지 않고, 보호 페이지 접근 시나리오가 자연스럽게 이어져야 하기 때문이다.

### 회원가입 화면에서도 라우팅 흐름 유지

- 로그인 과정에서 가입되지 않은 사용자라면 회원가입 화면으로 보내고, 이때도 `redirect`와 `reason`을 유지하도록 했다.
- 이유는 `보호 페이지 -> 로그인 -> 회원가입 -> 로그인 -> 원래 페이지` 흐름이 한 경로로 이어져야 사용자 이동이 단절되지 않기 때문이다.

### 로그아웃 이후 이동 흐름 정리

- 로그아웃 버튼을 통해 세션을 정리한 뒤 로그인 페이지로 이동시키고, `reason=logout`을 함께 전달하도록 만들었다.
- 이유는 로그아웃 후 어디로 이동할지 정책이 명확해야 하고, 사용자가 현재 상태 변화를 바로 이해할 수 있어야 하기 때문이다.

### 접근 불가 상황 안내 문구 추가

- 로그인 페이지에서 `reason=auth`일 때 "로그인 후 접근 가능한 페이지"라는 안내 문구를 보여주도록 했다.
- 이유는 단순 이동보다 이동 이유까지 보여주는 쪽이 라우팅 흐름을 이해하기 쉽고 사용자 경험도 더 좋기 때문이다.

### 라우트 구조와 컴포넌트 역할 분리

- `router.js`를 `publicRoutes`, `userRoutes`, `myPageRoutes`, `authRoutes`로 나눠 읽기 쉽게 정리했다.
- `MyPage.vue`를 별도 페이지 레이아웃으로 만들고, `MyPageUserInfo.vue`를 중첩 라우트용 자식 화면으로 분리했다.
- 인증 화면인 `Join.vue`도 `Common/Auth` 아래로 이동시켜 `Login.vue`와 같은 영역으로 맞췄다.
- 이유는 "공개 페이지", "인증 페이지", "사용자 상세 페이지", "내 페이지"의 책임이 섞이지 않게 하려는 목적이다.

## 3. 라우트 구조와 컴포넌트 역할 분리

### 라우트 구조 분리

- 라우트는 공개 영역, 사용자 상세 영역, 마이페이지 영역, 인증 영역으로 나눌 수 있다.
- 이번 작업에서는 `publicRoutes`, `userRoutes`, `myPageRoutes`, `authRoutes`로 분리해 라우트 구성을 읽기 쉽게 정리했다.
- 이렇게 나누면 어떤 페이지가 보호 대상인지, 어떤 흐름에 속하는지 구조만 봐도 파악하기 쉬워진다.

### 중첩 라우트와 역할 분리

- 중첩 라우트는 부모 화면 틀은 유지하고 내부의 `<RouterView />`만 바꾸는 구조다.
- `/users/:username`은 사용자 상세 영역, `/my-page`는 로그인 사용자 전용 영역으로 역할을 나눴다.
- `MyPage.vue`는 마이페이지의 부모 레이아웃 역할을 하고, `MyPageUserInfo.vue`는 그 안에 들어가는 자식 화면 역할을 담당한다.

### 인증 화면 역할 분리

- 인증 관련 화면은 `Common/Auth` 아래에 모아 `Login.vue`, `Join.vue`를 같은 영역으로 정리했다.
- 이렇게 하면 인증 페이지는 공통 흐름을 담당하고, 사용자 상세 페이지나 마이페이지와 역할이 섞이지 않게 된다.

## 4. 학습포인트

- 인증이 필요한 페이지는 라우트 메타 정보로 표시하고, 실제 제어는 전역 가드에서 처리하면 정책이 한곳에 모인다.
- 라우팅 흐름은 "보호 페이지 접근", "로그인으로 이동", "성공 후 복귀"처럼 사용자 이동 순서로 설계하면 훨씬 명확해진다.
- `redirect` 같은 정보를 query로 전달하면 흐름이 URL에 남아서 추적과 디버깅이 쉽다.
- 로그인, 회원가입, 로그아웃처럼 서로 이어지는 화면은 이동 이유와 목적지를 함께 넘겨야 흐름이 끊기지 않는다.
- 접근 제한 상황에서 이유를 보여주는 문구를 추가하면 단순한 차단보다 훨씬 친절한 UX가 된다.
- 라우트 구조와 컴포넌트 역할을 분리하면 어느 화면이 어떤 흐름에 속하는지 읽기 쉬워지고 유지보수도 쉬워진다.
